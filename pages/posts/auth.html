Building a Secure Authentication System with PHP

Learn how to build a production-ready authentication system in PHP with password hashing, secure sessions, CSRF protection, and brute-force prevention.

Johnmark Delacruz

The first authentication system I ever built stored passwords in plain text. It "worked" — until the day it very obviously didn't. If you are shipping anything real, security is not a feature you add later; it is the foundation the rest of your app stands on. This is the layer I wish I had built the first time around.

We will cover the four pieces that actually matter: hashing passwords the right way, hardening sessions, protecting against CSRF, and throttling brute-force attempts. No frameworks required — just PHP and a clear mental model.

1. Hash passwords, never store them

If your database is ever leaked, a properly hashed password is useless to an attacker. Use the built-in password_hash() function with the default PASSWORD_DEFAULT algorithm — it handles salts and future-proofing for you.

register.php 
<?php
// Always hash on the way in — never trust the raw value
$email = filter_var($_POST['email'], FILTER_VALIDATE_EMAIL);
$password = $_POST['password'];

$hash = password_hash($password, PASSWORD_DEFAULT);

$stmt = $pdo->prepare('INSERT INTO users (email, password) VALUES (?, ?)');
$stmt->execute([$email, $hash]);

// password_verify() is the ONLY safe way to check a login
if (password_verify($password, $storedHash)) {
    // authenticated
}
Note · never roll your own Do not use md5() or sha1() for passwords. They are fast by design, which is exactly what makes them easy to brute-force. password_hash() uses bcrypt/argon2 and is deliberately slow.

2. Harden the session

A hash is only as safe as the session that remembers the user. Regenerate the session ID on login so a stolen cookie from before authentication is useless, and lock cookies down with strict flags.

session.php 
session_start([
    'cookie_secure'   => true,   // HTTPS only
    'cookie_httponly' => true,   // invisible to JS
    'cookie_samesite' => 'Lax',   // mitigates CSRF
]);

// Rotate the ID the moment a user authenticates
session_regenerate_id(true);

$_SESSION['user_id'] = $user['id'];
$_SESSION['auth_at'] = time();
Authentication is not "are you who you say you are?" — it is "can anyone impersonate you if they steal this token?" Design for the second question. — a lesson learned the expensive way

3. Stop CSRF with tokens

Cross-site request forgery tricks a logged-in browser into sending a request the user never intended. The fix is boring and effective: issue a random token per session and require it on every state change.

csrf.php 
// Generate once, store in session
if (empty($_SESSION['csrf'])) {
    $_SESSION['csrf'] = bin2hex(random_bytes(32));
}

// On the form
echo '<input type="hidden" name="csrf" value="' . $_SESSION['csrf'] . '">';

// On submit
if (!hash_equals($_SESSION['csrf'], $_POST['csrf'] ?? '')) {
    http_response_code(403);
    exit('Invalid request');
}

Why hash_equals matters

Use hash_equals() instead of === for token comparison. It runs in constant time, which prevents timing attacks that could otherwise leak the valid token byte by byte.

4. Rate limit brute force

Even great hashing can be worn down by unlimited attempts. Track failed logins per account and IP, then temporarily lock the doors. Here is a compact version using a small table.

throttle.php 
$maxAttempts = 5;
$window = 900; // 15 minutes

$stmt = $pdo->prepare(
    'SELECT COUNT(*) FROM login_attempts
     WHERE ip = ? AND created_at > (NOW() - INTERVAL ? SECOND)'
);
$stmt->execute([$ip, $window]);
$attempts = $stmt->fetchColumn();

if ($attempts >= $maxAttempts) {
    http_response_code(429);
    exit('Too many attempts. Try again later.');
}
Tip · log everything Every failed and successful login should land in an audit log. When (not if) something looks off, that trail is the only thing between you and a very bad day.

Putting it together

None of these pieces are hard on their own. The discipline is in doing all of them, every time, without exception. A login form is a promise to your users — that you took the time to protect what they trusted you with. Ship the hash, lock the session, sign the requests, and throttle the noise.

Build it once, build it right, and you can stop losing sleep over the part of your app that attackers go for first.
